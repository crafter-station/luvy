# Sharelove Run MVP Plan

## Product Decisions

- Build a mobile-first Next.js app where authenticated runners create public race pages and anonymous supporters leave audio messages.
- Public run URL shape is `/{runnerSlug}/{runSlug}`, matching examples like `/max/boston2026`.
- Use Clerk for runner authentication only; supporters submit anonymously with optional display name.
- Use Neon Postgres with Drizzle for runners, runs, and message metadata.
- Use Vercel Blob for audio file storage.
- Supporters can either record in-browser or upload an existing audio file.
- Supporters choose `public` or `private` visibility at submission time.
- Runner creates a run from one authenticated form: runner slug, run slug, title, race date/time, timezone, optional location.
- Race time controls playback availability.
- Public messages are playable on the public page only after race time.
- Private messages are never playable publicly; they are playable only in the authenticated runner playlist after race time.
- Before unlock, show locked waveform-style cards instead of exposing playable audio.
- Runner moderation is delete-only. Do not allow runner to change a supporter message from private to public.
- Upload limits: 2 minutes and 15 MB per message.
- Message lists support `createdAt` ascending and descending sorting.
- Runner playlist supports both individual play and a "play all" queue.
- Support light and dark mode.

## Existing Codebase Constraints

- App is a minimal Next.js `16.2.6` App Router project under `src/app`.
- Next 16 uses `proxy.ts`, not `middleware.ts`.
- Dynamic route `params` and route handler context params must be awaited.
- The repo uses Tailwind v4 CSS-first config in `src/app/globals.css`.
- Existing shadcn/base-ui style components live in `src/components/ui`.
- `next-themes` and `@clerk/nextjs` are already present in `package.json`, but neither is wired into the app yet.
- No database, Drizzle, Vercel Blob, API routes, server actions, or audio components currently exist.
- Use existing `cn()` from `src/lib/utils.ts` and preserve the established Tailwind token approach.

## Data Model

Create Drizzle schema in `src/db/schema.ts` and database client in `src/db/index.ts`.

Tables:

- `runners`
- `runs`
- `messages`

`runners` fields:

- `id` uuid primary key default random
- `clerkUserId` text unique not null
- `slug` text unique not null
- `displayName` text nullable
- `createdAt` timestamp with timezone default now not null
- `updatedAt` timestamp with timezone default now not null

`runs` fields:

- `id` uuid primary key default random
- `runnerId` uuid references `runners.id` on delete cascade not null
- `slug` text not null
- `title` text not null
- `raceStartsAt` timestamp with timezone not null
- `raceTimezone` text not null
- `location` text nullable
- `createdAt` timestamp with timezone default now not null
- `updatedAt` timestamp with timezone default now not null
- unique index on `runnerId + slug`

`messages` fields:

- `id` uuid primary key default random
- `runId` uuid references `runs.id` on delete cascade not null
- `visibility` text enum/check: `public` or `private`
- `supporterName` text nullable
- `blobUrl` text not null
- `blobPathname` text not null
- `contentType` text not null
- `sizeBytes` integer not null
- `durationSeconds` integer not null
- `originalFilename` text nullable
- `createdAt` timestamp with timezone default now not null
- `deletedAt` timestamp with timezone nullable

Indexes:

- unique `runners.slug`
- unique `runs.runnerId + runs.slug`
- `messages.runId + messages.createdAt`
- `messages.runId + messages.visibility + messages.createdAt`

## Auth And Access Control

- Wrap `src/app/layout.tsx` with Clerk provider and theme provider.
- Add `suppressHydrationWarning` to `<html>` for `next-themes`.
- Add Clerk catch-all sign-in and sign-up pages.
- Add `src/proxy.ts` to protect `/dashboard/:path*`.
- Keep `/`, `/sign-in`, `/sign-up`, `/{runnerSlug}/{runSlug}`, and `POST /api/messages` public.
- Treat proxy as convenience only. Every server action and protected route handler must call `await auth()` from `@clerk/nextjs/server` and verify ownership.
- Use `404` for unauthorized audio access where possible to avoid leaking private message existence.

## Validation Rules

Create schemas/helpers in `src/lib/validations.ts` and `src/lib/slugs.ts`.

Slug rules:

- lowercase letters, numbers, hyphens only
- 3 to 40 chars
- no leading or trailing hyphen
- reject reserved top-level slugs: `api`, `dashboard`, `sign-in`, `sign-up`, `_next`, `favicon.ico`

Run form:

- runner slug required
- run slug required
- title required
- race date required
- race time required
- IANA timezone required
- location optional

Message upload:

- `runnerSlug` and `runSlug` required
- visibility must be `public` or `private`
- supporter name optional, max 80 chars
- audio file required
- max size 15 MB
- max duration 120 seconds
- content type must be common audio type, including `audio/webm`, `audio/mp4`, `audio/mpeg`, and `audio/wav`

For MVP, enforce duration in the browser and store client-reported duration while enforcing size/type on the server. Avoid adding ffmpeg in the first pass unless strict server duration validation becomes mandatory.

## Timezone And Unlock Logic

- Store `raceStartsAt` as UTC timestamp.
- Store `raceTimezone` as the selected IANA timezone string.
- Convert date/time/timezone on the server when creating a run.
- Use a shared helper in `src/lib/dates.ts` for formatting race time and checking unlock state.
- Centralize visibility/playback rules in `src/lib/messages.ts`.

Playback rules:

- Public visitor can play a message only when it is public, not deleted, and current time is after `raceStartsAt`.
- Public visitor never sees playable private messages.
- Authenticated runner owner can play public and private messages only after `raceStartsAt`.
- Runner can view message metadata before race time but playback is disabled.
- Deleted messages are excluded everywhere except any internal query needed for audit/debugging.

## Upload Flow

- Build `supporter-message-form.tsx` as a client component.
- Build `audio-recorder.tsx` using `MediaRecorder` for in-browser recording.
- Build `audio-upload-field.tsx` as fallback/manual upload.
- Client validates duration and size before submitting.
- Submit `FormData` to `POST /api/messages`.
- `src/app/api/messages/route.ts` validates form data, finds the run, uploads to Vercel Blob, inserts message metadata, and returns JSON.
- Use generated/random blob paths, not supporter filenames.
- Recommended blob path shape: `runs/{runId}/messages/{randomId}.{ext}`.

## Audio Access Flow

Create `GET /api/messages/[messageId]/audio`.

- Load message, run, and runner.
- Check race-time unlock.
- If unauthenticated, allow only public messages after unlock.
- If authenticated, allow only the owning runner after unlock.
- Return `404` for private/publicly unauthorized, pre-race, deleted, or missing messages.
- Redirect or stream to the Vercel Blob URL only after authorization.
- Do not render raw Blob URLs in locked public states or private public states.

Note: if Vercel Blob objects are public by unguessable URL only, this protects app-level discovery but not access to a leaked URL. If hard private-object security is required, evaluate Vercel Blob private access support or switch audio bytes to a signed URL storage provider later.

## Server Actions

- `createRun(formData)`
- require Clerk auth
- lazily create or update runner row using Clerk user ID
- validate slugs and reserved words
- validate race date/time/timezone

- redirect to `/dashboard/runs/{runId}`

## Addendum: Reduce Race Setup Friction

### Goal

Simplify new race creation so runners do not need to understand handles, slugs, timezones, or event metadata before getting a shareable page.

### Product Changes

- Remove runner handle input from the new race form.
- Generate the runner handle from Clerk user data the first time a runner profile is created.
- Let runners edit their handle later from the dashboard.
- Remove run slug input from the new race form.
- Generate the run slug from the selected event preset or race title.
- Let runners edit the run slug later from the run dashboard.
- Warn clearly that changing either handle or run slug changes the public link and requires resharing.
- Make race title the first field in the creation flow.
- Add event presets/autocomplete for common races.
- Add built-in presets:
  - `Lima 2026 42K`: `2026-05-24`, `06:00`, `Lima, Peru`, `America/Lima`

- Users can still edit any prefilled field before creating the run.
- Timezone should be selected from available IANA timezone values, not typed freely.
- For custom/manual races, default timezone to the browser/server default currently used by the form.
